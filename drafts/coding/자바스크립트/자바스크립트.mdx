1. ECMA 표준은 TC39 위원회에서 제정한다. 이딜은 약 50~100명으로 구성되며, 브라우저를 만드는 조직(모질라, 구글, 애플)과 하드웨어 제조사(삼성 등) 같이 웹에 투자를 많이 하는 회사 출신이 주를 이룬다.

2. JS의 버전은 여러개일까? JS의 공식 버전은 단 하나다. 2000년대 초반, MS가 JS를 불완전하게 리버스 엔지니어링해 JSscript 라는 언어를 만들었을 땐 여러 버전의 JS가 있긴 했지만, 현시대에서 여러 버전이 있다고 주장하는 것은 부정확하다. ECMA 표준 JS만을 JS라고 여긴다.

3. JS 엔진 제조사들은 명세서 개정안을 반영했을 때 기존에 잘 보이던 컨텐츠가 깨지는 경우, 명세서 개정안을 자사 엔진에 반영하지 않겠다는 결정을 내리게 된다. 이런 불일치가 발생하면 TC39 위원회는 종종 기존 결정을 철회하고 명세서를 웹에 맞춘다. 배열 contains(➝ 충돌로 인해 includes로 변경), flatten(➝ 충돌로 인해 flat으로 변경)이 대표적이다.

4. JS처럼 보이지만 사실은 웹 자체의 API인 것: alert, setTimeout, fetch, getCurrentLocation, getUserMedia 등
5. 마찬가지로 node.js 역시 자체의 API를 가지고 있다: fs, path, crypto 등
6. 그리하여 특정 환경에서만 쓸 수 있는 API가 있는 것이다: 클라이언트(브라우저) / 서버(node.js)
7. alert는 JS가 호출하지만, 이를 추가한 건 공식JS(ECMA)가 아니라 브라우저 엔진 회사들이다.
8. 하위호환성 보장: 이전 버전에서 사용된 문법은 시간이 지나도 유효성을 보장받는다. 명세서에 한번이라도 들어간 내용은 절대제거하지 않는다는 원칙. 다만 예외는 있다. 그러나 아주 드물게 일어나고 그 효과도 미미하다.
9. 반대로 상위호환성이란 새로 나온 문법을 사용했을 때 구형 JS에서 오류를 내지 않는 경우인데, 경험적으로 이건 아니라는 것을 안다.
10. 한편 HTML과 CSS는 상위 호환성을 보장하지만 하위 호환성은 보장하지 않는다.
11. 새로운 문법을 사용했을 때 옛 브라우저가 이를 지원하지 않는 경우? 트랜스파일을 통해 구버전의 코드도 대비한다.
12. 트랜스파일은 JS 커뮤니티에서 만든 용어로, 한 형태에서 다른 형태로 소스 코드를 변환해주는 것을 의미한다. 컴파일이 소스코드를 머신코드로 변환한다면, 트랜스파일은 소스코드를 또다른 소스코드로 변환하는 것. 바벨, swc 같은 트랜스파일러는 실행 환경에서 아직 지원하지 않는 문법이나 API를 수동으로 조정해 코드 가독성이 떨어지지 않게 한다.
13. 개발자의 소스코드 ➝ 바벨/swc의 트랜스파일 ➝ 웹팩 등의 번들링 ➝ JS 엔진의 AST 파싱 ➝ 이진 바이트 코드(JIT 컴파일러) ➝ 프로그램 실행
14. 상용 환경에 배포되는 JS 코드 대부분은 트랜스파일 처리된다. 그렇기 때문에 엄격모드를 쓰지 않았더라도 사실상 엄격모드가 기본으로 자리 잡았다고 볼 수 있다.

15. == 비교 연산자는 타입 강제 변환 동등 비교 연산자다. 비교 대상이 모두 문자열인 경우를 제외하고는 숫자 타입으로 타입을 변환해 비교를 진행한다. 문자열일 경우에는 알파벳순으로 문자열을 비교한다.

 16. `this`에 대한 일반적인 오해: 함수 자기 자신을 가리킨다, 메서드가 속한 인스턴스를 참조한다.

```ts
Object.create()
Object.is()
function.call(ctx, param1, param2)
function.apply(ctx, [param1, param2])
function.bind(ctx)
```
17. Object.create(oldObj): oldObj를 프로토타입으로 삼는 새로운 오브젝트 생성

18. 프로토타입 위임 패턴, 클래스 상속 패턴, 함수형 프로그래밍
19. 클래스는 객체 프로퍼티, 모듈은 스코프 변수를 통해 데이터와 메서드에 접근
20. 클래스에서는 모든 데이터와 메서드가 공개
